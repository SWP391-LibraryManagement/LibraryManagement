# Phase 3 Performance Report - 2026-07-19

## Decision

The Phase 3 polish keeps the existing route/API contracts and lazy-loads the
top-level React page modules. The initial JavaScript entry bundle is now below
the previous Vite advisory size; route guards remain in the entry module so
authorization behavior does not move into page chunks.

## Frontend build measurement

Both measurements are raw minified JavaScript bytes from `npm.cmd --prefix
frontend run build` in the isolated Phase 3 worktree.

| Metric | `origin/main` baseline | Phase 3 branch | Change |
| --- | ---: | ---: | ---: |
| Initial entry JS | 999,203 bytes | 320,688 bytes | -67.9% |
| JavaScript asset count | 1 | 57 | Route-level chunks added |
| Total JS across all assets | 999,203 bytes | 1,018,070 bytes | +1.9% |

The total is slightly larger because each route chunk carries its own module
boundary overhead, but the first page no longer downloads every operational
screen. The largest and entry asset after the change is `index-Dt3UORuG.js`.

## API timing measurement

Command:

```powershell
npm.cmd run phase3:performance
```

Environment: local deterministic in-memory E2E repositories, Node.js built-in
`fetch`, and bcrypt cost `10` (the approved minimum). The command creates
synthetic member/librarian identities, warms the endpoint, then records 30
login and 50 `/api/auth/me` samples. Tokens and identities are never printed.

| Operation | Samples | p50 | p95 target | Observed p95 | Result |
| --- | ---: | ---: | ---: | ---: | --- |
| Valid login | 30 | 67.56 ms | < 1,000 ms | 84.43 ms | PASS in documented local environment |
| Session validation (`/auth/me`) | 50 | 1.36 ms | < 50 ms | 2.06 ms | PASS in documented local environment |

The output is generated by `scripts/phase3-performance.js` and covered by
`tests/performance/phase3-performance.test.js`. It reports nearest-rank p50/p95
values, sample counts, bundle bytes, and limitations.

## Validation

- `node --test tests/performance/phase3-performance.test.js`: 3/3 passed.
- `npm.cmd --prefix frontend test`: 151/151 passed.
- `npm.cmd --prefix frontend run lint`: passed.
- `npm.cmd --prefix frontend run build`: passed with no entry-chunk warning.
- `npm.cmd run test:e2e`: the existing synthetic golden path remains the L4 UI
  regression check and is recorded in the Phase 3 user-testing record.

## Boundaries

- The timing harness uses in-memory repositories and therefore excludes Azure
  SQL connection latency, SQL query plans, and public-network latency.
- The login sample uses bcrypt cost 10 but does not prove a production Azure
  App Service p95 under concurrent load.
- SMTP/provider latency and real inbox delivery were not measured.
- JavaScript total bytes are not a production transfer-size metric; gzip/Brotli
  transfer sizes depend on the hosting CDN configuration.
- Further bundle splitting can remain a future optimization if real browser
  telemetry shows a need; the Phase 3 entry-chunk advisory is resolved.
