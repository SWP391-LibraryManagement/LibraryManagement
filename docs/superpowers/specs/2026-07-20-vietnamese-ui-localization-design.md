# Vietnamese UI Localization and Typography Design

Date: 2026-07-20

Status: IMPLEMENTED - PR #58 MERGED; APPLICATION-BASELINE AND LOCAL RECONCILIATION EVIDENCE RECORDED

Implementation evidence: `.sdd/reviews/vietnamese-ui-localization-validation-2026-07-20.md`.

## 1. Goal

Make all frontend-generated user-facing content Vietnamese by default while preserving
the existing library workflows, role permissions, API contracts, and business rules.
Improve Vietnamese glyph rendering with a consistent font pairing:

- `Be Vietnam Pro` for body text, controls, tables, and navigation.
- `Noto Serif` for page headings and other editorial emphasis.

Common technical terms approved by the user, including `Email`, `OTP`, and `Barcode`,
remain unchanged.

## 2. Scope

Included:

- Public browse and homepage content.
- Login, registration, forgot-password, and OTP flows.
- Member, librarian, and admin navigation and operational pages.
- Buttons, headings, table headers, placeholders, accessible labels, empty states,
  loading states, success notices, confirmation dialogs, and error messages generated by
  the frontend.
- User-facing labels for role and status codes, such as `Admin` -> `QuáșŁn trá» viĂȘn` and
  `AVAILABLE` -> `CĂł sáș”n`, without changing the underlying codes.
- Document language metadata and page title.
- Shared font tokens and scattered page-level font declarations.

Excluded:

- Database migrations and API contract changes.
- Translation of book titles, author names, email addresses, barcode values, or any
  user-entered/content-owned text.
- Changes to authentication, authorization, borrowing, reservation, fine, notification,
  or reporting business rules.
- A runtime language switcher or English translation catalog.

## 3. Proposed Architecture

### 3.1 Copy catalog

Add a lightweight frontend copy catalog at `frontend/src/i18n/vi.js` for reusable labels,
actions, roles, navigation groups, and common state messages. This is intentionally not a
full i18n framework because the approved product scope is Vietnamese-only.

Page-specific wording remains close to its component when that makes the workflow easier
to understand, but repeated terms use the catalog so that wording stays consistent.

### 3.2 Display labels

Add `frontend/src/utils/uiLabels.js` for presentation-only mappings. The functions accept
raw enum/status values and return Vietnamese labels. Existing business logic continues to
compare raw values such as `PENDING`, `AVAILABLE`, `BORROWED`, and `INACTIVE`.

### 3.3 API errors

Extend the existing API error helpers, primarily
`frontend/src/api/apiErrorMessages.js`, so known backend error codes resolve to actionable
Vietnamese messages. Unknown or unsafe server messages use a contextual Vietnamese
fallback rather than exposing raw technical English. Authentication and token-clearing
behavior remain unchanged.

### 3.4 Typography

Update `frontend/index.html` with `lang="vi"`, a Vietnamese page title, and the font
loading declarations. The CSS layer will define shared font variables and use fallbacks
that support Vietnamese if the remote font request is unavailable.

Replace page-level declarations of `Inter`, `Lato`, `Playfair Display`, `Times New Roman`,
and similar one-off families where they are used for the application UI. Preserve
monospace declarations for technical identifiers and code-like values.

## 4. Data and Rendering Flow

1. Backend responses continue to contain their canonical codes and source data.
2. API helpers normalize known errors into Vietnamese user-facing copy.
3. View models retain raw status values for filtering and business decisions.
4. `uiLabels` maps raw values to display text at the presentation boundary.
5. Components render Vietnamese labels and safe fallback states.
6. User-owned data is rendered as received and is never machine-translated.

This boundary prevents localization from changing API behavior or business rules.

## 7. Out of Scope and Risks

- If Google Fonts cannot be reached in a deployment environment, fallback fonts will be
  used; the implementation must not fail or hide content.
- Proper names and catalog metadata may still contain English because they are source data,
  not interface copy.
- A future English/ Vietnamese switch would require a separate approved design and should
  not be inferred from this work.
