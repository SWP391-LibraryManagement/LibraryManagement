# Constitution — Library Management System

# Version: 0.1.0

# Status: DRAFT

## 1. Project Purpose

This project is a Library Management System for SWP391.  
The system helps librarians and administrators manage books, members, borrowing, returning, overdue fines, and reports.

## 2. Development Method

The team follows Hybrid Spec-Driven & Agent-Driven Development.

- Spec-Driven Development is used for core requirements, business rules, database design, API contracts, and security-sensitive features.
- Agent-Driven Development is used to support implementation, testing, documentation, and refactoring.
- No core feature should be implemented without an approved SPEC.md.

## 3. Source of Truth

The source of truth for each feature is:

.sdd/specs/feat-{name}/SPEC.md

Implementation must follow:

1. SPEC.md
2. PLAN.md
3. TASKS.md
4. `.agents/AGENTS.md`
5. `.agents/CLAUDE.md`

If code conflicts with SPEC.md, the code is considered wrong unless the spec is updated and approved.

## 4. Core Business Rules

- A book cannot be borrowed if its available quantity is 0.
- A member cannot borrow more than the allowed borrowing limit.
- A member with overdue books or unpaid fines may be restricted from borrowing.
- Every borrow and return transaction must be recorded.
- Fine calculation must be traceable and testable.
- Admin actions must be logged when they affect books, members, borrowing, returning, or fines.

## 5. Specification Rules

Each core feature must have:

- SPEC.md
- CONTEXT.md
- PLAN.md
- TASKS.md
- CHANGELOG.md

A SPEC.md must include:

- Business context
- Actors
- Preconditions
- Main flow
- Business rules
- Acceptance criteria
- Edge cases
- Out of scope
- Non-functional requirements
- Traceability

## 6. Code Rules

- Code must be simple, readable, and maintainable.
- Do not add features outside the current SPEC.md.
- All user input must be validated.
- Database operations must use ORM or parameterized queries.
- No hardcoded passwords, API keys, tokens, or secrets.
- Business logic must not be hidden inside UI code.

## 7. Testing Rules

- Core business rules must have tests.
- Borrow book, return book, and fine calculation must have test cases.
- Existing tests must pass before merging.
- Code generated by AI must be reviewed by a human before commit or merge.

## 8. Git Rules

Branch naming:

- docs/{name}

- fix/{bug-name}
- refactor/{module-name}

Commit convention:

- docs:
- feat:
- fix:
- refactor:
- test:
- chore:

## 9. AI Usage Rules

AI/agent may assist with:

- Drafting specs
- Reviewing specs
- Breaking specs into tasks
- Generating code from approved tasks
- Writing tests
- Refactoring
- Documentation

AI/agent must not:

- Implement features without SPEC.md
- Add out-of-scope functionality
- Modify security-sensitive logic without human review
- Create or expose secrets
- Change database schema without updating related spec/ADR

## 10. Definition of Done

A feature is done only when:

- SPEC.md is approved.
- PLAN.md and TASKS.md are completed.
- Code satisfies all acceptance criteria.
- Required tests pass.
- No secrets are committed.
- Documentation is updated.
- Human review is completed.
